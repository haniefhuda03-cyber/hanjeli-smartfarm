## 1.1 Executive Summary

Hanjeli SmartFarm is an **IoT precision‑agriculture platform** for a single‑site farm. ESP32 field devices stream **soil telemetry** (pH, moisture, EC, NPK + N/P/K components, temperature) over **MQTT**; the NestJS backend persists it to **PostgreSQL + TimescaleDB** hypertables, evaluates an **automatic irrigation/fertigation engine**, and pushes **real‑time updates** to the Next.js dashboard over **Socket.IO**. Users monitor soil health, control irrigation (auto / scheduled / manual / emergency‑stop), manage devices, and administer accounts. Weather context comes from the **Open‑Meteo** API. Security includes **JWT access/refresh tokens, TOTP 2FA with recovery codes, Google OAuth, bcrypt hashing, pgcrypto‑encrypted TOTP secrets, and rate limiting**.

**Core objective:** keep soil parameters within user‑defined safe ranges by automating water and fertilizer pumps based on live sensor data, while giving operators full visibility and manual override.

## 1.2 User Personas & Roles

The backend recognizes exactly two roles (`UserRole = 'Admin' | 'Guest'`, `authenticated-user.interface.ts:1`; CHECK constraint `chk_users_role`, migration 15). New sign‑ups default to **Guest**.

| Persona | Role | Capabilities | Source of authorization |
|---------|------|--------------|--------------------------|
| **Admin** | `Admin` | Everything a Guest can do **plus**: list/create/update/delete users (`/users` admin routes); FE exposes **Akun Pengguna** + **Irigasi** tabs. | `@Roles('Admin')` on `users.controller.ts`; FE `adminOnlyIds = {users, irrigation}` (`dashboard-layout.tsx:51`) |
| **Guest** (standard farmer/operator) | `Guest` | Self‑service profile/security, devices CRUD (own), view sensors/monitoring/quality, weather, notifications, preferences. **Cannot** access `/users`. Irrigation tab is admin‑gated on the FE. | Default role on register (`auth.service.ts:76`) |
| **Device / Hardware (ESP32)** | n/a (not a DB user) | Publishes sensor telemetry & device status; receives irrigation commands; sends command ACKs over MQTT. Identified by **device `code`** (e.g. `WS004`), not by JWT. | `mqtt-sensor.handler.ts`, `mqtt-irrigation.handler.ts` |
| **Unauthenticated visitor** | none | Landing page (`/`), all `/auth/*` public pages, Google OAuth start. | Public routes |
| **Seed Admin** | `Admin` | Bootstrapped by `database/seed.ts` from `SEED_ADMIN_*` env (default `admin@hanjeli.local`). Refuses insecure placeholder passwords. | `seed.ts:20-34` |

> **Note on FE vs API authorization:** the FE hides the Irrigation tab from non‑admins, but the backend irrigation endpoints are guarded only by `JwtAuthGuard` (no `@Roles`). So **any authenticated user can call `/api/v3/irrigation/*` directly.** This is a test/security case (see §3).

## 1.3 End‑to‑End User Flows

Notation: **FE** = frontend page/action, **API** = REST endpoint, **DB** = persistence, **RT** = realtime, **MQTT** = broker.

### Flow A — Registration → Email Verification → Login

1. **FE** `/register` (`register/page.tsx`): user enters name/email/password/confirm + agrees to terms. Client password policy: ≥8 chars, uppercase, number required; confirm must match (`evaluatePassword`, submit disabled otherwise).
2. **API** `POST /auth/register` (throttle 5/min). Backend: lowercases email, rejects duplicate (incl. soft‑deleted, `withDeleted:true`) → `409 Conflict`; bcrypt‑hashes password (`BCRYPT_ROUNDS`, default 12); creates user `role='Guest'`, `email_verified=false`; creates an `auth_tokens` row (`purpose='email_verification'`, random 32‑byte token, HMAC‑SHA256 hashed, expiry `AUTH_EMAIL_TOKEN_EXPIRES_IN` def 24h); sends verification email via SMTP. Returns `{message, user}` (`auth.service.ts:59-92`).
3. **FE** redirects to `/register/verify-email?email=…`. Page shows "check email" + resend button (60s cooldown). If the URL also carries `?token=`, it auto‑calls verify.
4. **Email link** → `FRONTEND_AUTH_CALLBACK_URL?type=verify-email&token=…` → **FE** `/auth/callback` posts `POST /auth/verify-email {token}`.
5. **API** `verifyEmail`: finds usable token (not used/revoked/expired), sets `email_verified=true`, marks token `used_at`. `400` if already verified or token invalid.
6. **FE** redirects to `/login`. User logs in (`POST /auth/login`).
7. **API** `login`: validates credentials; **rejects unverified email** with `401 "Email belum diverifikasi"`; if `two_factor_enabled` → returns 2FA challenge (Flow C); else issues access+refresh JWT + user (`issueAuthTokens`).
8. **FE** stores tokens (`storeAuthSession` → localStorage) and routes to `/home`.

### Flow B — Google OAuth Login

1. **FE** `GoogleButton` → `window.location = /api/v3/auth/google` (`auth-shared.tsx:121`).
2. **API** `GET /auth/google` (passport google strategy) → Google consent → `GET /auth/google/callback`.
3. **API** `handleGoogleLogin`: matches by `google_id` then by email; links `google_id`/avatar to existing user; **sets `email_verified=true`**; creates a new Guest if none. If 2FA enabled → challenge; else tokens. Then **redirects** to `FRONTEND_AUTH_CALLBACK_URL?type=oauth&…` carrying either `access_token`+`refresh_token`+`token_type` or `requires_2fa`+`challenge_token` (`auth.controller.ts:135-155`).
4. **FE** `/auth/callback` parses query: stores session (and fetches `/users/me` to cache the user) → `/home`; or stores challenge token → `/login/verify-2fa`.

### Flow C — Two‑Factor (TOTP) Login Challenge

1. After password (or Google) success with 2FA on, **API** returns `{requires_2fa:true, challenge_token, expires_in}` (signed with `JWT_CHALLENGE_SECRET`, type `2fa_challenge`, def 10m) (`issueTwoFactorChallenge`).
2. **FE** `/login/verify-2fa`: stores challenge token (session storage), 6‑digit OTP input, 30s visual code timer. Submits `POST /auth/verify-2fa {challenge_token, token}`.
3. **API** `verifyTwoFactor`: verifies challenge token type, loads user, **pgp_sym_decrypt**s the stored TOTP secret, validates code via `otplib`. Success → issues access+refresh tokens.
4. Alternative: **FE** `/login/recovery` → `POST /auth/verify-recovery {challenge_token, code}`. Backend bcrypt‑compares against unused `recovery_codes`; on match marks `is_used=true` and issues tokens. Recovery code format `XXXX-XXXX`, normalized uppercase.

### Flow D — Enable / Disable 2FA (authenticated, in Profile)

1. **FE** `/profile` → 2FA panel toggle ON → `POST /auth/2fa/setup` (Jwt). Backend `generateSecret()`, **pgp_sym_encrypt**s secret into `users.two_factor_secret` (bytea), returns `{secret, otpauth_uri}` (NOT yet enabled).
2. **FE** renders QR from `otpauth_uri` via `QRCodeSVG` (secret never re‑sent to server). User scans, enters 6‑digit code → `POST /auth/2fa/enable {token}`.
3. **API** `enableTwoFactor`: validates TOTP, sets `two_factor_enabled=true`, **regenerates 8 recovery codes** (bcrypt‑hashed, replaces old), returns plaintext codes once. FE shows copy/download.
4. **Disable:** `DELETE /auth/2fa` (Jwt) → clears `two_factor_enabled` + `two_factor_secret`, deletes all recovery codes.

### Flow E — Forgot / Reset Password

1. **FE** `/forgot-password` → `POST /auth/forgot-password {email}` (throttle 3/min). Backend **always returns the same generic message** (anti‑enumeration); only if a user with a password exists, creates a `password_reset` token (def 30m) + emails reset link.
2. **Email** → `/auth/callback?type=reset-password&token=…` → **FE** redirects to `/reset-password?token=…`.
3. **FE** `/reset-password`: validates new password (≥8, etc.) → `POST /auth/reset-password {token, new_password}`.
4. **API** `resetPassword`: validates token, bcrypt‑hashes new password, marks token used, **revokes** all other active reset tokens. FE shows success → `/login`.

### Flow F — Live Sensor Monitoring (Home + Monitoring)

1. **MQTT** ESP32 publishes JSON to `hanjeli/sensor/<CODE>` (or legacy `hanjeli/<CODE>/sensor`).
2. **API** `MqttSensorHandler.handleSensorData`: resolves device by `code`; normalizes payload (accepts many key aliases: `ph|ph_level`, `moisture|soil_moisture`, `ec|soil_ec`, `npk|soil_npk`, `n/p/k|nitrogen/…`, `temp|temperature`, `ts`); sets device `status='online'`, `last_seen_at`; **inserts `sensor_telemetry`**; runs **irrigation engine**; invalidates sensor caches; **RT** broadcasts `sensor:realtime` to the user's room.
3. **FE** `/home` & `/monitoring`: React Query polls `GET /sensors/overview`, `/quality-score`, `/sensors/history`, `/sensors/trend`, `/sensors/stats` (intervals 60s/300s) **and** subscribes via `useSensorSocket` to `sensor:realtime`, merging live values into the 5 sensor cards. NPK card shows N/P/K bars. Monitoring also locally patches the trend chart with the live point.
4. **Quality score:** `GET /sensors/quality-score` computes a 0–100 score by penalizing each of 5 parameters against optimal/allowed ranges (`sensors.service.ts:205-268`).

### Flow G — Automatic Irrigation Engine (on each telemetry insert)

`IrrigationEngine.processSensorData` (`irrigation.engine.ts`):
1. Loads the user's `irrigation_config`. If none → no‑op.
2. **Emergency stop active** → force both water + fertilizer pumps OFF (`action='EMERGENCY_STOP'`, forced) and return.
3. If `active_mode !== 'auto'` → return.
4. **Auto evaluation** (`evaluateAutoMode`):
   - **Water:** hysteresis on soil moisture vs `water_min_threshold`/`water_max_threshold`. `< min` → pump ON; `> max` → OFF; in‑between → keep previous state (per‑device state map).
   - **NPK/fertilizer:** evaluates N, P, K vs their min/max. Status = `missing|low|normal|high`. If any **low** → fertilizer pump ON (and **water is forced off, fertilizer prioritized**). If any **high** → fertilizer held off. Status changes are logged (`warning` when low/high, `success` on return to normal), deduped via `lastNpkState`.
5. **Commands** are published via `MqttService.publishIrrigationCommand` to `hanjeli/irrigation/command` with `{action, mode:'auto', channel, speed, device_code, user_id, ts, request_id}`; an **ACK timeout** (def 10s) logs a warning activity if the ESP32 doesn't ACK. Only state *changes* are published (unless `force`).
6. Every command writes an `irrigation_activity_logs` row.

### Flow H — Manual / Mode Control (WebSocket, Irrigation page)

`/irrigation` (`useIrrigationSocket`) emits Socket.IO events handled by `IrrigationGateway`:
- **`irrigation:setMode`** `{mode, config?, deviceCode?}` → `updateConfig` (persists), then publishes commands: `manual` → per‑channel pump commands; `off` → STOP water+fertilizer; else → RESUME water. Broadcasts `irrigation:status`. Returns `{success, config}`.
- **`irrigation:manualToggle`** `{active, channel, speed?, deviceCode?}` → enforces **water/fertilizer mutual exclusivity**, persists, publishes the toggled pump command (+ forces the other pump off), logs activity, broadcasts status.
- **`irrigation:emergencyStop`** → sets `emergency_stop=true`, publishes EMERGENCY_STOP for both channels, emits `irrigation:emergency {active:true}` + status.
- **`irrigation:resume`** → `emergency_stop=false`, publishes RESUME, emits `irrigation:emergency {active:false}` + status.
- Schedules CRUD goes via REST (`POST/PUT/DELETE /irrigation/schedules`), not WS.
- **Speed is frozen at 100%** in the UI (`manualSpeed` state comment "PWM not available yet").

### Flow I — Scheduled Irrigation (cron, every minute)

`IrrigationCronService.handleCron` (`@Cron(EVERY_MINUTE)`): computes current day/`HH:mm:00`, selects active schedules for users whose `active_mode='scheduled'`, and on exact start/end match publishes to MQTT + writes an activity log. **See V3/V4 — the topic and join are suspect.**

### Flow J — Device Management

- **FE** `/profile` (Add Device panel) and `/home` device grid. **API** `GET/POST/PUT/DELETE /devices` (own devices only, scoped by `user_id`).
- Create: normalizes `code` (strip `#`, uppercase), rejects duplicate code (incl. soft‑deleted) → `409`; `status` defaults `offline`; `last_seen_at` set if created `online`.
- Delete: **blocked with `409`** if the device has any `sensor_telemetry` rows (history protection); otherwise soft‑delete.
- Device status also updated by MQTT `hanjeli/device/<CODE>/status` (`online|warning|offline` + message), broadcasting `device:status`.

### Flow K — Admin User Management

**FE** `/users` (admin only). **API**: `GET /users` (paginated, `q` search on name/email/role, `role` filter), `POST /users` (creates verified user), `PUT /users/:id`, `DELETE /users/:id` (soft‑delete; **admin cannot delete own account via this route** → `403`). FE fetches `limit=100` and paginates client‑side (5/page).

### Flow L — Self Account Deletion

**FE** `/profile` delete panel: type‑to‑confirm + password + (if 2FA) 6‑digit code → `DELETE /users/me {password, twoFactorToken?}`. Backend re‑authenticates password (if the account has one) and asserts the TOTP when 2FA is enabled, then soft‑deletes. FE clears session → `/login`.

### Flow M — Preferences

**API** `GET /preferences` returns `{preference, units, notification_prefs, sensor_thresholds}` and **lazily seeds defaults** on first read (5 units, 6 notif prefs (3 cat × 2 chan), 4 thresholds). `PUT /preferences` (language `id|en`, notifications_enabled), `PUT /preferences/units` (validates unit against allow‑list per parameter), `PUT /preferences/notification-prefs` (upsert per category+channel), `PUT /preferences/sensor-thresholds` (min<max enforced).

### Flow N — Notifications (realtime + bell)

`NotificationProvider` (global): on mount fetches `GET /notifications?limit=50` (falls back to `localStorage` key `hanjeli_notifications` on error) and opens a Socket.IO connection listening for **`notification:new`** → shows a Sonner toast (deduped within 2s) + rings the bell. Mark‑read/clear call `PATCH /notifications/:id/read`, `PATCH /notifications/read-all`, `DELETE /notifications/:id`, `DELETE /notifications`.

## 1.4 Edge Cases & Error Handling

### Hardware / sensor / connectivity

| Scenario | System behaviour | Evidence |
|----------|------------------|----------|
| Malformed MQTT JSON | Logged + dropped, no crash (`parsePayload` returns null). | `mqtt.service.ts:235-246` |
| Telemetry for unknown device `code` | Warned + ignored. | `mqtt-sensor.handler.ts:77-79` |
| Sensor message with no device code | Warned + ignored. | `mqtt-sensor.handler.ts:68-71` |
| Partial sensor payload (missing fields) | Missing values stored as `NULL`; NPK auto‑summed from N+P+K only if all present, else null. | `normalizeSensorPayload` |
| Out‑of‑range sensor values | DB **CHECK constraints** reject inserts (pH 0–14, moisture 0–100, EC/NPK/N/P/K ≥0, temp −50…80). A bad reading aborts that insert; handler logs error. | migration 7 |
| ESP32 never ACKs a command | `publishIrrigationCommand` timeout (def 10s) fires `onTimeout` → warning activity log. | `mqtt.service.ts:117-134` |
| MQTT broker down / disabled | `MQTT_ENABLED=false` → no connect; publish warns "client not connected" and no‑ops. Auto reconnect with exponential backoff (1s→30s). | `mqtt.service.ts:82-101,190-206` |
| Device goes offline | Only updated when ESP32 publishes `…/status` offline/warning; **no server‑side staleness sweep** of `last_seen_at`. | `handleDeviceStatus` |
| Redis unavailable | All cache ops degrade gracefully to in‑memory `Map` (AppCacheService) / fallback (Weather). Never throws. | `app-cache.service.ts`, `weather.service.ts` |
| Open‑Meteo API failure | Returns hardcoded **fallback** weather (24°C, "Cerah", `source:'fallback'`). | `weather.service.ts:52-62,189-205` |
| Delete device with history | `409 Conflict` (protect telemetry). | `devices.service.ts:84-94` |
| Delete device referenced by telemetry at DB level | FK `ON DELETE RESTRICT` on `sensor_telemetry.device_id`. | entity + migration 7 |

### Auth / security edge cases

| Scenario | Behaviour |
|----------|-----------|
| Login, wrong password / no password (OAuth‑only account) | `401 "Email atau password salah"` (generic). |
| Login, unverified email | `401 "Email belum diverifikasi"`. |
| Forgot password for unknown email or OAuth‑only account | Generic success message (no enumeration). |
| Reset/verify token reused, expired, or revoked | `400 "Token tidak valid atau sudah kedaluwarsa"`. |
| 2FA challenge token expired | `401` invalid/expired. |
| Recovery code reused | Marked `is_used`; reuse → `401 "Recovery code tidak valid"`. |
| Avatar upload non‑image / >2MB | `400` (allow PNG/JPG/WEBP/GIF, ≤2MB). |
| Change password on account with existing password without `currentPassword` | `401 "Kata sandi saat ini diperlukan"` / wrong → `401`. |
| Admin deletes own account via `/users/:id` | `403`. |
| Rate limit exceeded | `429` global throttle (10/60s default) + tighter per‑route throttles on auth. |
| Validation failure | `400` with structured `{message, error:'Bad Request', fields[], details[]}` (global ValidationPipe + `HttpExceptionFilter`). |
| WS connect without/invalid token | `auth:error` emitted then socket disconnected. |

### Frontend UI states

- **401 on any API call** → interceptor attempts one refresh via `/auth/refresh`; on failure clears session and (if not already on an auth page) redirects to `/login` (`apiClient.ts:90-126`, `client.ts:42-97`).
- **Loading / empty / error / success** states: dedicated components in `src/components/ui-states/*` plus per‑page `isLoading`/`error` from React Query (e.g. home history `historyError`).
- **Mobile/tablet (<1024px):** full maintenance overlay (V6).
- **Optimistic UI:** users list and avatar update local cache before invalidation.
- **Toast dedupe:** identical `type|title|description` within 2s suppressed (`notification-context.tsx:229`).

## 1.5 Acceptance Criteria (per feature)

**Auth — Registration/Verification**
- AC1: Duplicate email (active or soft‑deleted) → `409`; never creates a second row.
- AC2: New user persists with `role='Guest'`, `email_verified=false`, bcrypt hash present, and an `email_verification` token row exists.
- AC3: Login is blocked until `email_verified=true`.
- AC4: Verification token is single‑use and time‑bounded; reuse → `400`.

**Auth — Login/2FA/Recovery**
- AC5: Valid credentials with 2FA off → `{access_token, refresh_token, token_type:'Bearer', expires_in, user}`.
- AC6: 2FA on → `{requires_2fa:true, challenge_token, expires_in}` and **no** access token.
- AC7: Correct TOTP via `/auth/verify-2fa` → tokens issued; wrong code → `401`.
- AC8: Recovery code valid once; second use → `401`.
- AC9: Access token expires per `JWT_ACCESS_EXPIRES_IN` (def 15m); refresh per `JWT_REFRESH_EXPIRES_IN` (def 7d).

**Auth — Google OAuth**
- AC10: New Google user is created Guest with `email_verified=true`; returning user is linked by `google_id`/email; callback redirect carries correct params.

**Sensors / Monitoring**
- AC11: `/sensors/overview` returns latest reading + 5 parameter cards each with `{key,label,value,unit,status}`; NPK card includes `nitrogen/phosphorus/potassium`.
- AC12: `/sensors/quality-score` returns 0–100 `score` + `status` band (Buruk<60≤Cukup<80≤Baik<90≤Sangat Baik) + `reasons`.
- AC13: `/sensors/trend` & `/stats` read from the correct continuous aggregate (`sensor_hourly_stats` for `day`; `sensor_daily_stats` for `week/month`) and are scoped to the user's devices.
- AC14: `/sensors/history` is paginated (limit ≤100), date‑range validated (`from`≤`to` else `400`), scoped to user's non‑deleted devices.
- AC15: `/sensors/export` streams CSV with the documented header.
- AC16: A new telemetry insert pushes `sensor:realtime` to the owning user only.

**Irrigation**
- AC17: `GET /irrigation/config` auto‑creates a default `off` config if none.
- AC18: In `auto` mode, moisture `<water_min` starts the water pump; `>water_max` stops it; values within range keep state (hysteresis), with an MQTT command + activity log on each change.
- AC19: Any N/P/K below min → fertilizer pump ON and water suppressed; above max → fertilizer held; transitions logged.
- AC20: `emergency_stop=true` forces both pumps off regardless of mode and emits `irrigation:emergency`.
- AC21: Manual water and fertilizer cannot both be enabled (`400` on REST, mutual‑exclusion on WS).
- AC22: Threshold ranges enforce min<max (water 0–100), else `400`.
- AC23: Schedule requires ≥1 day and `start_time<end_time`, else `400`.

**Devices**
- AC24: Device codes are unique (normalized); duplicate → `409`.
- AC25: Deleting a device with telemetry → `409`.

**Users (Admin)**
- AC26: Non‑admin calling any `/users` admin route → `403`.
- AC27: Admin cannot delete own account via `/users/:id` → `403`.
- AC28: Self profile update changing email sets `email_verified=false`.

**Preferences**
- AC29: First `GET /preferences` seeds 5 units, 6 notif prefs, 4 thresholds.
- AC30: Invalid unit for a parameter → `400`; threshold min≥max → `400`.

**Cross‑cutting**
- AC31: All non‑auth, non‑root REST endpoints require a valid Bearer token → `401` otherwise.
- AC32: All error responses follow the global envelope `{success:false, statusCode, message, error, path, timestamp}` (+ `fields`/`details` on validation).
- AC33: CORS allows configured FE origins with credentials; helmet headers present.

## Appendix — Default values cheat‑sheet

- JWT: access 15m, refresh 7d, 2FA challenge 10m, email token 24h, reset token 30m.
- Bcrypt rounds 12. Recovery codes: 8 × `XXXX-XXXX`.
- Throttle: global 10/60s; register/login 5, refresh 30, forgot/resend 3, reset 5, 2FA routes 5.
- Irrigation defaults: `active_mode='off'`, water 30–80%, N/P/K 20–60 (npk 60–180), manual/fertilizer speed 100, `scheduled_behavior='manual'`.
- Quality bands: <60 Buruk, <80 Cukup, <90 Baik, else Sangat Baik.
- Sensor optimal ranges (status): pH 6–7.5, moisture 40–70%, EC 0.8–2, NPK 50–220, temp 20–30.
- Weather fallback: 24°C, "Cerah". Open‑Meteo coords def −7.19 / 106.61, TZ Asia/Jakarta.
- Avatar: ≤2MB, png/jpg/webp/gif. History page limit cap 100.
